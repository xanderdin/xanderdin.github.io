И кто сказал, что CI/CD только для софта?

2019-06-01
Переводы: EN

Недавно я гуглил на предмет использования CI/CD. Существует огромное количество статей на эту тему, но все они в основном о том, как применять эти практики в сфере разработки программного обеспечения. Неужели это единственная область, где эти подходы применимы? Конечно нет. Вот мой пример использования их в области, которая не относится к разработке программного обеспечения.

Ситуация

Представьте себе небольшую студию дизайна интерьеров с небольшой командой дизайнеров. Их работа заключается в том, чтобы предоставить клиентам идеи, образы и технические чертежи, которые в последствии могут использовать строители для превращения идей в реальность. Создание технических чертежей является обычно завершающим этапом большинства дизайн проектов, когда абстрактные идеи трансформируются в фактические размеры и спецификации.

В своём производстве команда использует свободные программные инструменты с открытыми исходными кодами. Конечно, они сталкиваются с трудностями когда дело касается совместной работы над одним и тем же проектом. В результате тех трудностей обычно только один дизайнер работает над завершающим этапом проекта, в то время как остальные не принимают участия в этом этапе. Несомненно, это очень непродуктивно и увеличивает общее время процесса, делая как клиентов, так и самих членов команды не сильно-то счастливыми.

Трудности исходят из того, как программное обеспечение, которое они используют, работает с листами чертежей. Каждый чертёж является отдельным файлом. Проект обычно состоит из 50 и более чертежей. И сборка всех тех чертежей, присваивание каждому листу своего номера, преобразование их в множество PDF файлов, и в конце-концов сборка их в единый PDF файл, всё это выполняется вручную, что является долгим по времени и потенциальным для появления ошибок (часы монотонной ручной работы). И пока все файлы не будут готовы и собраны, конечный результат нельзя легко увидеть, проверить и при необходимости скорректировать.

Но благодаря тому, что используется программное обеспечение с открытым исходным кодом, такие инструменты могут быть улучшены и дополнены. Кроме того, другие инструменты с открытым исходным кодом могут быть использованы для того, чтобы помочь в искоренении описанных трудностей и созданию условий для более простой и удобной процедуры производства технической документации.

Идея

В наше время из каждого угла слышно упоминание о практиках DevOps, организации CI/CD, эджайл методологиях в разработке программных продуктов. Почему бы не попробовать и не применить те же самые подходы в области, которая не связана с производством программного обеспечения? Давайте-ка попробуем посмотреть и представить, что можно сделать для описанной выше команды дизайнеров.

Цели:

  • Автоматизировать этап производства технический чертежей.
  • Сделать возможным видеть результат в любое время производственного этапа.
  • Обеспечить условия совместного сотрудничества для возможности вести работу параллельно.

Инструменты:

  • Git
  • Docker
  • Jenkins
  • enumsheets
  • dxf2pdf

Все перечисленные инструменты будут работать на Debian GNU/Linux (Stretch). Я подразумеваю о наличии подходящего компьютера, на котором Docker уже установлен и исправно работает.

Примечание: ниже описан лишь упрощенный пример того, что можно воплотить в реальности.

Примерный производственный процесс

Дизайнеры подготавливают технические чертежи и коммитят их в свои локальные git-репозитории. После подготовки каждого нового листа технического чертежа они пушат (git push) свою работу в основной git-репозиторий. Jenkins, в конвейере (pipeline) извлекает чертежи из основного git-репозитория, обрабатывая их с помощью скрипта enumsheets вставляет нумерацию и другие данные в их основные надписи. Далее, обрабатывает пронумерованные листы с помощью инструмента dxf2pdf, конвертируя множество листов в единый PDF файл. Готовый PDF файл затем копируется в общую директорию, доступную всем членам команды.

Для подготовки чертежей дизайнеры используют LibreCAD, а также следуют требованиям enumsheets в отношении основной надписи листов.

Воплощение

Для начала нужно создать Docker-образы для инструментов enumsheets и dxf2pdf. Для этого надо создать две директории где-нибудь в $HOME, одну для enumsheets, а другую для dxf2pdf. Также надо создать по два файла в каждой и этих директорий, Dockerfile и build.sh:

Dockerfile для enumsheets:

FROM python:3-alpine
WORKDIR /app
COPY enumsheets/enumsheets.py ./
COPY enumsheets/config.ini ./
COPY enumsheets/requirements.txt ./
RUN pip install --no-cache-dir --trusted-host pypi.python.org -r ./requirements.txt
ENV PATH=$PATH:/app
CMD ["sh"]

Скрипт build.sh для enumsheets:

#!/bin/sh

CWD=$(pwd)

# enumsheets v1.0.1 commit:
COMMIT="37789348564ae881acd54c8e3bc70cea5f8ebc65"

# clean
[ -d enumsheets ] && rm -rf enumsheets

# clone
git clone https://github.com/xanderdin/enumsheets.git

# use particular commit
cd enumsheets
git checkout -b "commit_${COMMIT}" "${COMMIT}"
cd "${CWD}"

# build
docker build ./ --tag=enumsheets

Требуется сделать build.sh исполнимым и запустить его:

chmod +x build.sh
./build.sh

Если сборка прошла успешно, мы получим Docker-образ для enumsheets. Для того чтобы убедится, что он существует, можно выполнить:

docker image ls enumsheets

Далее, нужно перейти в директорию для dxf2pdf, создать Dockerfile и build.sh со следующим содержимым.

Скрипт build.sh для dxf2pdf:

#!/bin/sh

CWD=$(pwd)

# dxf2pdf commit
COMMIT="d01efb3f33d4cc257fa0bb6d68098193cc03fa19"

# clean
[ -d ./app ] && rm -rf ./app
[ -d ./LibreCAD_dxf2pdf_standalone ] && rm -rf ./LibreCAD_dxf2pdf_standalone

# clone
git clone https://github.com/xanderdin/LibreCAD_dxf2pdf_standalone.git

# build dxf2pdf
cd ./LibreCAD_dxf2pdf_standalone
git checkout -b "commit_${COMMIT}" "${COMMIT}"
./prepare.sh
mkdir build
cd build
cmake ../LibreCAD
make -j 4
mv OUT "${CWD}/app"
cd "${CWD}"

# build docker image
docker build ./ --tag=dxf2pdf

Требуется сделать build.sh исполнимым и запустить его:

chmod +x build.sh
./build.sh

Если сборка прошла успешно, мы получим Docker-образ для dxf2pdf. Для того чтобы убедится, что он существует, можно выполнить:

docker image ls dxf2pdf

Директории для общего хранилища

Ради простоты этого примера я создам основной git-репозиторий на том же самом компьютере, на котором запущено и всё остальное. Мои директории для совместного доступа будут выглядеть так:

/home/000/artidea/proj      -->  основная директория
/home/000/artidea/proj/git  -->  основные git-репозитории для проектов
/home/000/artidea/proj/OUT  -->  директория для финальных файлов

Мой проект будет размещаться в директории 0001, поэтому нужно создть структуру директорий для этого проекта:

mkdir -p /home/000/artidea/proj/{git,OUT}/0001

Основной git-репозиторий

Теперь нужно инициализировать основной пустой git-репозиторий для проекта 0001:

cd /home/000/artidea/proj/git/0001
git init --bare

Jeinkins будет запускаться в Docker-контейнере и подключать в свою внутреннюю директорию /cicd директории наших проектов:

        Фактически                Jenkins
--------------------------       ---------
/home/000/artidea/proj/git  -->  /cicd/git
/home/000/artidea/proj/OUT  -->  /cicd/OUT

Для того чтобы заставлять Jenkins запускать конвейер (pipeline) после каждого git push в основной git-репозиторий, я буду использовать git-хук скрипт. Для моего проект 0001 необходимо создать хук-скрипт в директории /home/000/artidea/proj/git/0001/hooks/, файл с именем post-receive содержащий следующее:

#!/bin/sh
# Notify Jenkins about a push
curl http://artidea.home:8080/git/notifyCommit?url=file:///cicd/git/0001

Обратите внимание, что именем моего хоста на котором будет доступен Jenkins является artidea.home. Вам вероятно нужно изменить это имя в хук-скрипте в соответствии с вашим окружением. И не забудьте сделать этот скрипт исполнимым выполнив команду:

chmod +x /home/000/artidea/proj/git/0001/hooks/post-receive

Jenkins

Теперь пришло время запустить Jenkins в первый раз. Я буду использовать Jenkins из jenkinsci/blueocean Docker-образа. При первом запуске я рекомендую запускать контейнер Jenkins без отсоединения его от терминала, для того чтобы видеть его лог. Это потому, что во время своего первого запуска, Jenkins напечатает в своём логе первоначальный пароль администратора, который требуется для осуществления дальнейшей настройки посредством веб-интерфейса.

Итак, давайте-ка запустим Jenkins с помощью следующей команды:

docker run -ti --rm \
     -u root \
     -p 8080:8080 \
     -v jenkins-data:/var/jenkins_home \
     -v /var/run/docker.sock:/var/run/docker.sock \
     -v /home/000/artidea/proj/git/:/cicd/git \
     -v /home/000/artidea/proj/OUT/:/cicd/OUT \
     --name my-jenkins \
     jenkinsci/blueocean

Для последующих запусков Jenkins можно создать скрипт. Конечно, в реальных условиях необходимо создать скрипт для запуска Jeinkins во время загрузки системы. Но для этого примера я не будут этого делать, лишь ограничусь созданием простого скрипта и положу его где-нибудь в $PATH. Скрипт в сущности тот же самый как и команда выше, с той лишь разницей, что опции -ti заменены опцией -d для запуска Docker-контейнера в отсоединённом от терминала режиме.

my-jenkins.sh:

#!/bin/bash

docker run -d --rm \
     -u root \
     -p 8080:8080 \
     -v jenkins-data:/var/jenkins_home \
     -v /var/run/docker.sock:/var/run/docker.sock \
     -v /home/000/artidea/proj/git/:/cicd/git \
     -v /home/000/artidea/proj/OUT/:/cicd/OUT \
     --name my-jenkins \
     jenkinsci/blueocean

Когда Jenkins запускается впервые, нам необходимо его настроить. Запустим web-браузер и перейдем по адресу Jenkins, http://artidea.home:8080 в моём случае, ваш вероятно другой. Возьмём начальный пароль администратора Jenkins из его лога и вставим его в соответствующее поле в веб-интерфейсе.

Продолжим настройку посредством веб-интерфейса. В моём случае я установил все предложенные плагины.

Создадим пользователя Admin.

Зададим URL Jenkins.

И наконец, Jenkins готов к работе.

Продолжим настройкой нового конвейера для проекта 0001.

Введём название проекта (proj0001), выберем “Pipeline”, далее нажмём кнопку “OK”.

Для информативности, введём описание проекта.

Пометим “Do not allow concurrent builds” и “Poll SCM” чек-боксы. Не нужно ничего вводить в поле “Schedule”, потому что будем использовать git-хук для нотификации Jenkins о новых изменениях в git-репозитории. “Poll SCM” чек-бокс однако должен быть помечен.

В секции Pipeline установим значение поля “Defenition” в “Pipeline script from SCM”, значение поля “SCM” в “Git”, в поле “Repository URL” введём “file:///cicd/git/0001”.

Нажмём кнопку “Save” и получим готовый к работе конвейер.

Структура файлов и директорий поекта 0001

Теперь давайте создадим структуру директории проекта 0001. Нам понадобится какая-нибудь директория, где мы будем работать над нашим проектом. Можно создать ту директорию где-нибудь в своём $HOME. После этого надо клонировать основной git-репозиторий в ту директорию, перейти в клонированный репозиторий и выполнить первоначальную подготовку:

git clone /home/000/artidea/proj/git/0001
cd 0001

Нужно создать директории для конфигурации и листов:

mkdir config sheets

Нужно скопировать конфигурационный файл config.ini утилиты enumsheets из её GitHub репозитория, в директорию config нашего проекта и именовать его enumsheets.ini:

URL="https://raw.githubusercontent.com/xanderdin/enumsheets/37789348564ae881acd54c8e3bc70cea5f8ebc65/config.ini"
wget "$URL" -O config/enumsheets.ini

Также нужно отредактировать этот конфигурационный файл в соответствии с потребностями проекта.

Затем нужно создать Jenkinsfile в корне клонированной директории проекта 0001 со следующим содержимым:

#!/usr/bin/env groovy

pipeline {
    agent none
    stages {
        stage('Clean') {
            agent any
            steps {
                echo 'Removing old data...'
                sh 'rm -rf ${WORKSPACE}/enumerated_sheets*'
            }
        }
        stage('Enum') {
            agent {
                docker {
                    image 'enumsheets'
                }
            }
            steps {
                echo 'Enumerating sheets...'
                sh '''
                /app/enumsheets.py -c ${WORKSPACE}/config/enumsheets.ini ${WORKSPACE}/sheets/*.dxf
                '''
            }
        }
        stage('Print') {
            agent {
                docker {
                    image 'dxf2pdf'
                    args "--entrypoint=''"
                }
            }
            steps {
                echo 'Printing to PDF...'
                sh '''
                /app/dxf2pdf -o ${WORKSPACE}/All.pdf ${WORKSPACE}/enumerated_sheets/*.dxf
                '''
            }
        }
        stage('Save') {
            agent any
            steps {
                echo 'Saving output file...'
                sh 'cp ${WORKSPACE}/All.pdf /cicd/OUT/0001/'
            }
        }
    }
}

Обратите внимание, что первая строка (shebang) в этом Jenkinsfile не требуется для Jenkins, но полезна для текстовых редакторов, чтобы помочь им применить правильную подсветку синтаксиса во время его редактирования.

Вот так выглядит конечная структура файлов и директорий:

0001
    ├── config
    │   └── enumsheets.ini
    ├── Jenkinsfile
    └── sheets

Теперь нужно добавить все изменения в git и выполнить git push для сохранения наших изменений в основном git-репозитории, путем выполнения следующего набора команд в клонированной директории проекта 0001:

git add .
git commit -m 'Initial commit'
git push

С этого момента все дизайнеры в команде могут клонировать себе основной git-репозиторий, начать создавать новые листы чертежей, сохраняя их в директории sheets, отправляя свою работу в основной git-репозиторий и постоянно получая результат совместной работы в виде единого PDF файла (All.pdf) внутри общедоступной директории /home/000/artidea/proj/OUT/0001/, готового для обзора и ранних исправлений в любое время в течении целого этапа производства технической документации.

Обратите внимание, что если вдруг после добавления первых чертежей и передачи их в основной git-репозиторий Jenkins не запустит конвейер, тогда вероятно потребуется запустить тот конвейер в этот раз вручную из веб-интерфейса. После этого каждая новая отправка изменений в основной git-репозиторий должна срабатывать как ожидается, если всё настроено правильно.

Вы можете поэкспериментировать с выше описанным окружением используя DXF файлы примеров из GitHub репозитория проекта enumsheets, которые располагаются в директории example/orig. Можно загрузить их и один за другим класть в директорию sheets проекта 0001, отправляя изменения в основной git-репозиторий и наблюдая как изменяется файл /home/000/artidea/proj/OUT/0001/All.pdf после каждого git push и прохода конвейера.

Заключение

После организации окружения по описанной конфигурации поставленные цели были достигнуты. Теперь каждый член команды может в любой момент наблюдать результаты совместной работы, вносить требуемые исправления на раннем этапе и работать параллельно с другими участниками команды над одним и тем же проектом. Автоматизация устранила процесс ручной сборки и многократно уменьшила время от начала работы над проектной документацией до получения результата (от часов ручной работы до нескольких секунд автоматической сборки). И этот пример показывает, что не только команды занятые производством программных продуктов могут получать выгоду от использования CI/CD практик, но и другие команды, которые заняты в других производственных областях также. Очевидно, что существует множество таких же команд, нуждающихся в подобных улучшениях. Почему бы не найти и не помочь им?
