Who said CI/CD is only for software?

2019-06-01
Translations: RU

Recently I googled for CI/CD use cases. There is a huge amount of articles on the subject but all of them are mostly about how to apply those practices in a software development environment. Is this the only way of how those practices and techniques can be used? Certainly not. And here’s my example of applying them in a non-software production.

Plot

Imagine a small interior design studio with a small team of designers. Their work is to provide clients with ideas, visuals and technical drawings suitable for construction work when implementing those ideas in reality. Creating technical drawings is usually the final stage of any design project when abstract ideas transform into actual dimensions and specifications.

The team uses free open source tools for their production work. Of course, they face some difficulties when it comes to collaboration on the same project. As a result of those difficulties usually only one designer works on the final stage of the project while others are not participating in that stage. Certainly, it is counterproductive and increases overall time of the process, making clients and team members unhappy.

The difficulties come from the way of how the software they are using works with drawings. Every drawing is a single file. A project usually consists of 50 or more drawings. And assembling all those drawings, first assigning all sheets numbers, then converting them to multiple PDF files, and finally to a single PDF file, all this doing manually is a very time consuming and error-prone procedure (hours of monotonuas manual work). And before all files are ready and assembled the final result cannot be easily seen, checked and corrected.

But thanks to the open source nature, those tools can be improved and extended. And even other open source tools can be used to help eliminate those difficulties making the procedure more straightforward and convenient.

Idea

Nowadays we hear from every corner about DevOps practices, CI/CD streamlines, agile methodologies in software production. Why not try and apply those in a non-software production environment? Let’s try and see what can be done for the described team of designers.

Goals:

  • Automate technical drawings production stage.
  • Make result visible at any time of the production stage.
  • Provide collaborative environment allowing work in parallel.

Tools:

  • Git
  • Docker
  • Jenkins
  • enumsheets
  • dxf2pdf

All tools will be running on Debian GNU/Linux (Stretch). I’m assuming a suitable computer is already provisioned, Docker is installed and running.

Note: this is just a simplified example of what can be done in reality.

Example production workflow

Designers draw technical drawings and commit them to their local git repositories. After every newly drawn sheet they push their work to main git repository. A pipeline in Jenkins fetches drawings from main git repository, processes them with enumsheets script inserting sheet numbers and other data to their title blocks, then processes enumerated sheets with dxf2pdf tool getting a single PDF file as output. The output PDF file is then copied to a shared directory available to all team members.

Designers use LibreCAD for making drawings and follow enumsheets requirements for drawings’ title blocks.

Implementation

First, let’s create Docker images for enumsheets and dxf2pdf tools. Create two directories somewhere in your $HOME, one for enumsheets and other for dxf2pdf. Create two files in each of those directories, Dockerfile and build.sh:

Dockerfile for enumsheets:

FROM python:3-alpine
WORKDIR /app
COPY enumsheets/enumsheets.py ./
COPY enumsheets/config.ini ./
COPY enumsheets/requirements.txt ./
RUN pip install --no-cache-dir --trusted-host pypi.python.org -r ./requirements.txt
ENV PATH=$PATH:/app
CMD ["sh"]

build.sh script for enumsheets:

#!/bin/sh

CWD=$(pwd)

# enumsheets v1.0.1 commit:
COMMIT="37789348564ae881acd54c8e3bc70cea5f8ebc65"

# clean
[ -d enumsheets ] && rm -rf enumsheets

# clone
git clone https://github.com/xanderdin/enumsheets.git

# use particular commit
cd enumsheets
git checkout -b "commit_${COMMIT}" "${COMMIT}"
cd "${CWD}"

# build
docker build ./ --tag=enumsheets

Make build.sh executable and run it:

chmod +x build.sh
./build.sh

After successful build we’ll get a Docker image for enumsheets. To see if it exists, run:

docker image ls enumsheets

build.sh script for dxf2pdf:

#!/bin/sh

CWD=$(pwd)

# dxf2pdf commit
COMMIT="d01efb3f33d4cc257fa0bb6d68098193cc03fa19"

# clean
[ -d ./app ] && rm -rf ./app
[ -d ./LibreCAD_dxf2pdf_standalone ] && rm -rf ./LibreCAD_dxf2pdf_standalone

# clone
git clone https://github.com/xanderdin/LibreCAD_dxf2pdf_standalone.git

# build dxf2pdf
cd ./LibreCAD_dxf2pdf_standalone
git checkout -b "commit_${COMMIT}" "${COMMIT}"
./prepare.sh
mkdir build
cd build
cmake ../LibreCAD
make -j 4
mv OUT "${CWD}/app"
cd "${CWD}"

# build docker image
docker build ./ --tag=dxf2pdf

Make build.sh executable and run it:

chmod +x build.sh
./build.sh

After successful build we’ll get a Docker image for dxf2pdf. To see if it exists, run:

docker image ls dxf2pdf

Directories for shared storage

For the simplicity of this example setup I’m going to create main git repository on the same computer where all other stuff runs. My directory structure for shared access will be as follows:

/home/000/artidea/proj      -->  base directory
/home/000/artidea/proj/git  -->  git main repositories for projects
/home/000/artidea/proj/OUT  -->  output files storage

My example project will be located in 0001 directory, so let’s create directory structure for this project:

mkdir -p /home/000/artidea/proj/{git,OUT}/0001

Main git repository

Now let’s initialize main bare git repository for 0001 project:

cd /home/000/artidea/proj/git/0001
git init --bare

Jeinkins will be run in a Docker container and its directory mappings will map our projects directories to its inner /cicd directory as follows:

          Actual                  Jenkins
--------------------------       ---------
/home/000/artidea/proj/git  -->  /cicd/git
/home/000/artidea/proj/OUT  -->  /cicd/OUT

I’m going to use git hook script in order to trigger Jenkins pipeline run after each push to main git repository. For my 0001 project I need to create a hook script in /home/000/artidea/proj/git/0001/hooks/ directory, a file named post-receive with the following contents:

#!/bin/sh
# Notify Jenkins about a push
curl http://artidea.home:8080/git/notifyCommit?url=file:///cicd/git/0001

Note, my hostname at which Jenkins will be available is artidea.home. Change this name in the hook script according to your environment. Don’t forget to make that script executable by running:

chmod +x /home/000/artidea/proj/git/0001/hooks/post-receive

Jenkins

Now it’s time to run Jenkins for the first time. I’m going to use Jenkins from jenkinsci/blueocean Docker image. For the first time I recommend running Jenkins container without detaching it from terminal in other to see its log. That’s because during its first run Jenkins will print to its log the initial admin password required to proceed with setup by using web-UI.

So let’s run Jenkins with the following command:

docker run -ti --rm \
     -u root \
     -p 8080:8080 \
     -v jenkins-data:/var/jenkins_home \
     -v /var/run/docker.sock:/var/run/docker.sock \
     -v /home/000/artidea/proj/git/:/cicd/git \
     -v /home/000/artidea/proj/OUT/:/cicd/OUT \
     --name my-jenkins \
     jenkinsci/blueocean

For next Jenkins runs we can create a script. Of course, in production environment one should create a script for starting Jeinkins during system boot. But for this example I’m not going to do that, just create a simple script and put it somewhere on the $PATH. The script is almost the same as the run command above except the -ti options are replaced by -d option in order to run Docker container in a detached mode.

my-jenkins.sh:

#!/bin/bash

docker run -d --rm \
     -u root \
     -p 8080:8080 \
     -v jenkins-data:/var/jenkins_home \
     -v /var/run/docker.sock:/var/run/docker.sock \
     -v /home/000/artidea/proj/git/:/cicd/git \
     -v /home/000/artidea/proj/OUT/:/cicd/OUT \
     --name my-jenkins \
     jenkinsci/blueocean

When Jenkins is running for the first time we need configure it. Let’s run web-browser and go to Jenkins address, http://artidea.home:8080 in my case, yours may differ. Find Jenkins initial admin password in its log and insert it into web-UI field.

Proceed configuration through web-UI. For my setup I installed all suggested plugins.

Create Admin user.

Configure Jenkins URL.

And finally Jenkins is ready.

Proceed with creating a new item pipeline for the 0001 project.

Enter project name (proj0001), select “Pipeline”, then press “OK” button.

Enter project description for clarity.

Check “Do not allow concurrent builds” and “Poll SCM” checkboxes. Do not enter anything in “Schedule” field because we’re using git hook to notify Jenkins about new git pushes. “Poll SCM” should be checked though.

In Pipeline section set “Defenition” field value to “Pipeline script from SCM”, “SCM” field value to “Git”, “Repository URL” field value to “file:///cicd/git/0001”.

Click “Save” button and this pipeline is ready.

Project 0001 files and directory structure

Now let’s create 0001 project directory structure. We need some directory where we’ll be doing our work on our technical drawings. Create that directory somewhere in your $HOME. Then clone main repository to that directory, go to cloned directory and make initial preparations:

git clone /home/000/artidea/proj/git/0001
cd 0001

Create directories for configuration and sheets:

mkdir config sheets

Copy enumsheets config.ini file from enumsheets GitHub repository to config directory in our project directory and name it enumsheets.ini:

URL="https://raw.githubusercontent.com/xanderdin/enumsheets/37789348564ae881acd54c8e3bc70cea5f8ebc65/config.ini"
wget "$URL" -O config/enumsheets.ini

Edit it according to your needs and set required configuration options.

Next, create Jenkinsfile in the root of 0001 project cloned directory containing:

#!/usr/bin/env groovy

pipeline {
    agent none
    stages {
        stage('Clean') {
            agent any
            steps {
                echo 'Removing old data...'
                sh 'rm -rf ${WORKSPACE}/enumerated_sheets*'
            }
        }
        stage('Enum') {
            agent {
                docker {
                    image 'enumsheets'
                }
            }
            steps {
                echo 'Enumerating sheets...'
                sh '''
                /app/enumsheets.py -c ${WORKSPACE}/config/enumsheets.ini ${WORKSPACE}/sheets/*.dxf
                '''
            }
        }
        stage('Print') {
            agent {
                docker {
                    image 'dxf2pdf'
                    args "--entrypoint=''"
                }
            }
            steps {
                echo 'Printing to PDF...'
                sh '''
                /app/dxf2pdf -o ${WORKSPACE}/All.pdf ${WORKSPACE}/enumerated_sheets/*.dxf
                '''
            }
        }
        stage('Save') {
            agent any
            steps {
                echo 'Saving output file...'
                sh 'cp ${WORKSPACE}/All.pdf /cicd/OUT/0001/'
            }
        }
    }
}

Note, the first (shebang) line in this Jenkinsfile is not required by Jenkins but is useful for text editors to help them apply right syntax highlighting.

Here’s resulting files and directories structure:

0001
    ├── config
    │   └── enumsheets.ini
    ├── Jenkinsfile
    └── sheets

Add all changes to git and push them to main git repository by running the following in the 0001 project cloned directory:

git add .
git commit -m 'Initial commit'
git push

From this point every designer on the team can clone or pull from the main git repository, draw a new drawing saving it in sheets directory, pushing work back to main git repository and instantly getting collaborative result as a single PDF file (All.pdf) inside shared /home/000/artidea/proj/OUT/0001/ directory, ready for review and early changes at any time during the whole production stage.

Note, if after your first drawings push to main git repository Jenkins not starts the pipeline, then start that pipeline manually from Jenkins web-UI. After that each new push to main git repository should work as expected (if everything is configured correctly).

You can experiment with this setup by using example DXF files from enumsheets project GitHub repository available in example/orig directory. Download them and put them one by one in any order to sheets directory of 0001 project, pushing changes to main git repository and watching how /home/000/artidea/proj/OUT/0001/All.pdf file changes with every git push and pipeline run.

Conclusion

By using described setup the original goals were met. Now everyone on the team can see at any stage what the result looks like, make required corrections early in the process and work with other team members in parallel on the same project. Automation eliminated manual assembly process and hugely decreased the time from drawing to result (from hours of manual work to a few seconds of automated assembly). And this example shows how not only software development teams can benefit from CI/CD techniques, but teams in other production environments also. Obviously, there are plenty of them waiting for similar improvements. Why not find and help them?
